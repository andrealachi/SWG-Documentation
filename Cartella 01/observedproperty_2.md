
# Observed Properties (GLOSIS) — Introduction

This repository provides a consolidated table of **observed properties** for soil description and analysis. The properties are sourced from the **GLOSIS code lists** (Global Soil Information System), published in the GLOSIS Linked Data documentation. These codelists supply controlled identifiers (IRIs) and human‑readable labels for observable properties and, where available, their **coded values**, enabling reliable interoperability across tools and projects.[1]

## Source and scope
- **Authoritative source**: *GLOSIS code lists*. The public page enumerates the codelists for observable properties (e.g., weathering, porosity, physico‑chemical properties) and the vocabularies of their values.[1]
- **Context**: the codelists are part of the GLOSIS ontology network, which is openly documented and maintained. The modules reuse Semantic Web standards such as **SKOS** for vocabularies and **SOSA/SSN** for observations and measurements.[2]

## What you will find here
- A **properties table** listing each observed property with a unique identifier and a link to the official codelist entry. The included properties **correspond to those on the “GLOSIS observable properties codelist” page**.[1]
- For properties that expose a set of **coded values** (classes, categories, scales), references are provided so that datasets can align with the official vocabulary.[1]

## Integration in the GeoPackage schema
- The **`observedproperty`** table holds the imported **GLOSIS properties** (identifiers, names, definitions/URIs, and descriptions).
- Where present, the **coded values** associated with each property are imported into the **`codesit`** table (vocabulary entries), using the **codelist collection IRI** as the key.
- **Join key**: `observedproperty.definition` ⇄ `codesit.collection` (the property’s *definition* IRI equals the *collection* IRI used for its values). This mirrors how GLOSIS links properties and value vocabularies in the Linked Data model.[1]

## Why this is useful
Reusing the same identifiers (IRIs) and GLOSIS codelists ensures **interoperability**, **traceability**, and **reusability**: datasets are easier to cross‑walk, and analytical pipelines can dereference authoritative definitions when needed.[2]

## Versioning and provenance
- The codelists page documents **releases** of the module (e.g., *v1.5.1* with version metadata). If you refresh the table, always record the **retrieval date** and, when available, the **module version**.[1]
- For an overview of the modules (main, observation, codelists, procedures, units, etc.), refer to the GLOSIS documentation entry page.[2]

## How to update this table
1. **Check** the index page of the **GLOSIS code lists** regularly for changes or new properties/values.[1]
2. **Update** the properties in **`observedproperty`** and the values in **`codesit`**, keeping the `definition` ⇄ `collection` join **unchanged**, and **do not modify the `guid` field** of existing properties (the **`observedproperty` GUID** must remain stable over time).
3. **Preserve** the original **IRIs** and keep descriptive labels for readability and machine‑readable provenance.[2]
